# 为什么需要 MarkItDown Skill

AI 要理解一份文档，第一步通常不是修改它，而是把其中的内容稳定地提取出来。

现实中的输入却很杂：报告可能是 PDF，方案可能是 Word，数据可能在 Excel，演示材料可能是 PowerPoint，参考资料也可能来自 HTML、JSON、图片、音频、压缩包或 YouTube。每种格式都有不同的解析方式、依赖和失败模式。如果 Agent 每次遇到文件都临时选择工具、安装库、编写转换代码，不仅浪费时间，也容易得到结构不一致、无法复用或未经验证的结果。

`markitdown` Skill 的作用，就是为这些“先把内容读出来”的任务提供一个统一入口。

它基于 Microsoft 的 [MarkItDown](https://github.com/microsoft/markitdown)，将多种文件和 URL 转换成适合大语言模型读取的 Markdown。转换重点不是复刻页面外观，而是保留对理解有帮助的标题、列表、表格、链接和元数据，使后续的总结、问答、检索、分类和 RAG 入库可以使用一致的文本表示。

## 为什么仅仅安装 MarkItDown 还不够

一个 Python 库提供的是能力，Skill 提供的是使用这项能力时需要遵循的判断和流程。

只把 MarkItDown 安装到环境里，Agent 仍然需要临时决定：

- 当前任务究竟是提取内容，还是编辑原文件；
- 应该使用宽泛的 `convert()`，还是权限范围更小的 `convert_local()`、`convert_stream()` 或 `convert_response()`；
- 某种格式缺少解析依赖时该安装哪个 extra；
- 多个文件如何批量转换，并避免同名输出互相覆盖；
- 什么时候可以启用第三方插件；
- 图片、扫描 PDF 或音频何时需要 OCR、转录或模型辅助；
- 是否值得把文件发送到 Azure 等付费云服务；
- 如何检查输出不是空文件，并确认关键结构确实被保留下来。

这些问题不是库 API 本身能够替 Agent 回答的。`markitdown` Skill 把它们固化为可重复执行的工作流，并附带经过测试的批量转换脚本，让 Agent 不必在每次任务中重新发明同一套流程。

## 相比其他文档 Skill，它的优势是什么

`markitdown` 和 Word、PowerPoint、Excel、PDF 等专用 Skill 解决的是不同阶段的问题。

| 任务 | 更适合的 Skill | 原因 |
| --- | --- | --- |
| 读取、总结或检索一个文档 | `markitdown` | 快速得到结构统一、适合模型消费的 Markdown |
| 把一批混合格式资料送入 RAG | `markitdown` | 用同一个入口处理多种格式，并支持递归批量转换 |
| 修改 Word 内容并保持样式 | Word/文档 Skill | 需要理解和写回文档结构，而不是有损提取 |
| 调整 PowerPoint 页面和视觉设计 | 演示文稿 Skill | 需要保持布局、图形和母版信息 |
| 修改 Excel 公式、格式或工作表 | 表格 Skill | 需要保留单元格类型、公式和工作簿行为 |
| 编辑、合并或精确渲染 PDF | PDF Skill | 需要页面级操作与视觉校验 |

最简单的判断方式是：

> 从文件中读取内容，优先考虑 `markitdown`；向文件中写入或修改内容，使用格式专用 Skill。

这种分工避免了两个常见问题。第一，使用重量级文档编辑流程去完成简单的文本提取，增加不必要的运行成本。第二，误把有损的 Markdown 转换当作高保真编辑工具，最终丢失布局、批注、公式或其他原生格式信息。

## 相比临时脚本，它的优势是什么

直接运行一条 CLI 命令适合单次、明确的输入，但真实任务经常会扩展成批量处理、错误恢复和结果验证。Skill 自带的 `scripts/convert.py` 提供了统一的批处理行为：

- 同时接受多个文件或整个目录；
- 递归处理目录并保留相对路径；
- 在输出名中保留原扩展名，降低同名碰撞风险；
- 默认不覆盖已有结果；
- 单个文件失败时保留已经完成的转换；
- 默认关闭第三方插件；
- 对不存在、解析失败或输出为空的输入给出明确错误。

这让批量转换成为可审计、可重复的操作，而不是一次性的 shell 循环。

## 相比通用文本提取器，它的优势是什么

纯文本提取通常只能得到字符流。对于大语言模型而言，结构本身也是信息：标题表达层级，列表表达并列关系，表格表达字段与值的对应关系，链接保留来源上下文。

MarkItDown 的目标正是尽可能把这些结构转成 Markdown。与无格式文本相比，这通常能减少后续清洗工作，并让模型更容易完成：

- 长文档摘要和章节问答；
- 表格与正文的联合分析；
- 文档切分和向量化；
- 多来源资料的统一检索；
- 从压缩包或混合文件集合中建立知识库。

它同时保持输出足够轻量，不要求下游系统理解每一种原始文件格式。

## Skill 额外提供的安全边界

文档转换不是纯计算操作。MarkItDown 会以当前进程的权限读取文件或访问 URL，插件和云端解析还可能把内容交给第三方代码或服务。

因此，这个 Skill 明确要求：

- 本地文件优先使用 `convert_local()`；
- 不可信 URL 应由调用方控制网络请求，再使用 `convert_response()`；
- 不默认启用第三方插件；
- 不把凭证写入命令、输出或仓库；
- 发送敏感文件到 Azure 或模型服务前先确认授权；
- 保留原始文件，并抽查 Markdown 与源内容是否一致。

这些约束让 Agent 不只是“能够转换”，还知道在什么权限范围内转换，以及什么时候应该停下来请求用户确认。

## 适合使用的典型场景

当用户提出下面这些请求时，可以优先调用 `markitdown` Skill：

- “把这份 PDF 转成 Markdown。”
- “读取这个 Word 文档并总结重点。”
- “把这一批方案和表格整理成可检索文本。”
- “提取这个演示文稿里的标题、正文和表格。”
- “把资料目录转换后用于 RAG 入库。”
- “读取这个网页或 YouTube 内容，供后续分析。”
- “为什么 MarkItDown 无法处理这个 DOCX？”

如果目标是修改源文件、保留精确排版、添加批注、维护公式或重新生成页面，则应该切换到对应的格式专用 Skill。

## 它不能替代什么

`markitdown` 不能替代高保真文档编辑，也不能保证所有复杂版式都被完整还原。扫描件可能需要 OCR，图片和图表可能需要视觉模型，复杂表格可能需要人工抽查，某些格式还依赖额外组件。

它也不应该被视为自动可信的摄取管道。转换成功只说明工具产生了输出，不代表内容完整、正确或适合直接进入生产知识库。Skill 因此要求对代表性章节、表格、链接和空白输出进行验证，并如实报告提取较弱或不支持的内容。

## 总结

`markitdown` Skill 的价值不只是“把文件转成 Markdown”。它把多格式读取、工具选择、依赖处理、批量执行、安全限制和结果验证组织成一个稳定工作流。

它与其他文档 Skill 的关系不是竞争，而是分层协作：`markitdown` 负责把内容从不同载体中提取出来，格式专用 Skill 负责高保真地创建、编辑和验证原始文件。前者让 Agent 更快获得可以分析的统一输入，后者保证需要写回时不丢失原生格式能力。
