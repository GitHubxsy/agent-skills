# AI 写代码越快，我们越需要认真看代码

AI 正在快速改变软件开发。过去需要几天完成的重构，现在可能在一个下午生成。代码的生产速度提高了，这是好事。

但它也带来了一个容易被忽略的问题：**生成代码的速度，正在超过团队理解代码的速度。**

当一个改动能在几分钟内产生几百行代码时，人很容易把“它能运行”“测试通过”“AI 解释过了”当成“我们已经理解并接受了它”。这三件事并不等价。代码一旦进入仓库，它就不再属于生成它的 AI，而属于未来要维护、升级、排障和承担后果的人。

因此，在 AI 时代，代码检视并没有过时。恰恰相反，它正在从一种工程习惯变成团队保持项目控制权的关键机制。

这也是 `explain-diff-for-human-review` 这个 Skill 出现的原因。

## 项目失控，往往不是从一个明显的错误开始

我们通常把代码风险理解为 bug：数组越界、空指针、竞态条件、错误的业务判断。AI 当然可能写出这些问题，但项目失控往往以更安静的方式发生。

一个新的配置参数悄悄改变了默认行为；一个为了复用而增加的抽象模糊了原有边界；一个“临时”兼容逻辑成为永久负担；一个测试通过了主流程，却没有覆盖故障恢复；一组为了方便验证而提交的证书，同时把 CA 私钥带进了仓库历史。

这些改动未必会立刻导致测试失败。它们甚至可能有充分的局部理由。但如果团队没有看清它们带来的系统影响，就会一点点失去对项目的掌握：

- 不再确定某个行为为什么存在；
- 不敢删除看似多余的逻辑；
- 不知道配置之间有哪些隐含约束；
- 无法判断测试通过是否真的代表可以发布；
- 出现事故时，只能重新考古整个调用链。

AI 会放大这个问题，因为它极大降低了“再加一点代码”的成本。过去，开发者会因为实现成本而克制；现在，一句提示词就能生成新的分支、包装层、配置项和测试。代码变多并不一定意味着能力同比增长，却一定意味着未来需要理解的表面积变大了。

所以真正需要守住的，不只是代码正确性，而是团队对代码的**可解释性、可追责性和可维护性**。

## 为什么只看原始 diff 已经越来越困难

原始 diff 是精确的，但它不是为理解系统而设计的。它告诉我们哪些行增加、删除或移动，却通常不会主动回答下面这些问题：

- 这次修改试图解决什么问题？
- 改动前的系统模型是什么？
- 为什么选择这个边界，而不是另一个边界？
- 用户、运维和其他模块会观察到什么变化？
- 哪些配置仍然兼容，哪些组合现在会失败？
- 测试证明了什么，又没有证明什么？
- 这里是明确缺陷、设计取舍，还是需要持续观察的残余风险？

资深开发者可以沿着符号、调用方、测试和历史提交逐步还原这些信息。但当 diff 规模扩大、跨越多个模块，或者 reviewer 同时面对多个 AI 生成的改动时，这种还原过程会消耗大量注意力。

结果往往有两种。第一种是检视停留在命名、格式和局部实现上，真正的系统风险被跳过。第二种是 reviewer 因为理解成本太高，最终只确认“CI 是绿的”。

这不是 reviewer 不负责，而是输入材料不适合人类建立完整心智模型。

## 我们需要的不是 AI 替人批准代码

面对检视压力，一个直觉方案是再引入一个 AI，让它自动 review AI 生成的代码。自动审查当然有价值，它适合发现常见缺陷、缺失测试、危险 API 和风格问题。但如果最终结果只是另一个“通过”或“不通过”，人的控制权仍然没有回来。

因为代码检视不只是找错。它还包含许多只有项目所有者才能作出的判断：

- “任意一个集群可用就算 Ready”，是否符合我们的可用性承诺？
- 为了兼容旧配置保留一条路径，是否值得长期维护？
- 为了本地验证提交固定证书，是否接受它被误用的安全风险？
- 指标只提供总量，是否足以支持生产排障？
- 这个抽象虽然正确，是否符合团队未来的演进方向？

这些问题没有脱离上下文的标准答案。AI 可以搜集证据、指出取舍、模拟失败场景，但最终决定必须由真正承担项目责任的人作出。

因此，`explain-diff-for-human-review` 的定位非常克制：**它不替人检视，而是为人准备检视。**

## 这个 Skill 如何帮助人做检视

这个 Skill 接收一个明确的代码改动范围，可以是单个提交、提交区间、分支比较、PR/MR、暂存区或工作区。它不会只读取 diff，然后逐行改写成自然语言，而是围绕改动建立检视所需的上下文。

### 1. 先还原改动前后的系统模型

要理解一个变化，首先要知道原来是什么样。Skill 会检查修改前的代码、相关调用方、测试、配置和文档，说明原有职责边界，再描述新的结构和数据流。

例如，一个 agent 从连接单个服务集群扩展为连接多个独立集群。原始 diff 可能只显示新增了地址列表和一个聚合对象，但真正需要 reviewer 确认的是：

- 一个地址代表一个实例，还是一个集群入口？
- 集群内实例发现是否仍然独立？
- 一个集群故障会不会污染另一个集群的状态？
- readiness 是要求全部集群健康，还是任意一个健康？

当这些关系被画成结构图、整理成修改前后模型时，reviewer 才能判断设计是否符合真实需求。

### 2. 按行为和责任组织改动

文件顺序通常不是最佳阅读顺序。一个功能可能同时修改配置、运行时编排、指标和测试。如果按文件逐个解释，读者需要自己在脑中重新拼装完整行为。

Skill 会按责任重新组织内容，例如：

1. 配置如何进入系统；
2. 运行时如何创建和维护状态；
3. 故障与重试如何发生；
4. 指标如何反映新模型；
5. 测试如何证明关键行为。

这种组织方式让 reviewer 看到的是一条完整因果链，而不是一堆彼此分离的文件变化。

### 3. 把影响面显式列出来

很多风险来自“我们根本没想到要看那里”。因此报告会使用影响矩阵，主动检查配置、API 或协议、运行时行为、并发、安全、可观测性、部署、兼容性和测试。

影响矩阵不是为了制造形式感，而是防止检视被局部实现吸走注意力。即使某个表面没有变化，明确写出“协议未改变”，也能帮助 reviewer 确认改动边界。

### 4. 区分缺陷、设计决定和残余风险

自动审查容易把所有问题都写成同一种“建议”。但人工检视需要知道每个发现意味着什么。

报告会区分：

- **已确认缺陷**：存在明确失败路径，应在合入前修复；
- **设计决定**：实现本身可能正确，但需要项目 owner 明确认可；
- **残余风险**：当前可以接受，但应记录边界或安排后续工作；
- **提示**：帮助理解代码，不代表必须修改。

每个发现都包含证据、可能的失败场景和建议的处理决定。这样 reviewer 不会只看到一句抽象的“这里有风险”，而能知道风险如何发生，以及自己需要决定什么。

### 5. 把“测试通过”拆成可审计的证据

AI 生成的说明最危险的习惯之一，是把存在于代码里的测试写成“测试已通过”。这个 Skill 要求严格区分三类证据：

1. 本次实际执行并观察到结果的测试；
2. diff 中存在，但本次没有执行的测试；
3. 根据风险建议补充的验证。

这种区分看起来细微，却直接关系到报告是否可信。reviewer 可以快速判断结论建立在真实执行、已有记录，还是合理推断之上。

### 6. 输出适合人阅读和留档的独立 HTML

报告最终生成一个自包含 HTML 文件。样式全部内嵌，不依赖远程脚本、字体或文档平台，可以直接打开、打印或导出 PDF。

HTML 报告提供固定目录、风险等级、架构图、影响矩阵、可折叠代码与验证命令，以及需要 reviewer 明确确认的清单。它不是为了比 Markdown 更华丽，而是为了让复杂改动拥有一个稳定、可扫描、可分享的检视界面。

同时，报告中的代码链接会根据当前仓库 remote 识别 GitHub、CodeHub、GitLab、Gitee 或其他平台。只有在能够可靠确定永久链接规则时才生成 URL；无法识别时，就退化为仓库相对路径、提交哈希和行号，而不是猜测地址。

## 一个具体例子：能用，不代表适合合入

在一次真实检视中，最新提交只增加了本地验证所需的证书文件。它没有修改一行运行时代码，证书与私钥全部匹配，证书链也能通过验证。从“后续验证是否方便”的角度看，这个提交完全实现了目标。

但报告进一步指出：提交中包含两把未加密 CA 私钥和四把端实体私钥；根证书还是缺少标准 CA 扩展的 X.509 v1 证书。任何继续信任这些 CA 的环境，都可能被持有仓库内容的人伪造身份。

这正是人工检视不可替代的地方。

“是否方便验证”是事实；“是否接受公开测试 CA”“是否必须在 README 中声明边界”“是否应该修复跨平台证书生成流程”是项目决策。Skill 可以把事实和风险摆到同一张桌面上，但最终必须由人决定哪些代价可以接受。

## 它不能替代什么

这个 Skill 不能替代熟悉业务的人，也不能保证发现所有 bug。它不知道团队尚未写下的承诺，不知道某个客户依赖的隐含行为，也无法替负责人承担一次错误合入的后果。

它同样不能替代测试、静态分析、安全扫描和真实环境验证。它所做的是把这些证据组织起来，指出证据之间的空白，让人更容易看见自己还不知道什么。

换句话说，它不是裁判，而是一份更好的案卷。

## AI 时代，人的角色不是少写几行代码

AI 让开发者从大量机械实现中解放出来，但这不意味着人的角色变轻了。人的角色正在向更高层移动：定义问题、选择边界、确认风险、维护一致性，并对系统长期结果负责。

代码检视正是这些责任汇合的地方。

未来的团队可能会让 AI 完成大部分初始实现，也可能让 AI 自动运行测试、生成文档和提出审查意见。但只要代码仍然影响真实用户、真实数据和真实业务，我们就需要有人能回答：

> 我们是否理解这次改变？我们是否接受它带来的行为和代价？如果它在半年后出问题，我们是否知道从哪里开始处理？

`explain-diff-for-human-review` 希望做的，就是让这些问题更容易被提出，也更容易被认真回答。

AI 可以帮助我们写得更快。检视让我们仍然知道，项目正在走向哪里。
