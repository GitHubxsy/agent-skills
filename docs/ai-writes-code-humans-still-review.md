# AI 写代码越快，我们越需要认真看代码

AI 正在快速改变软件开发。过去需要几天才能完成的功能，现在可能一个小时就能生成。代码生产变快了，这是好事。

但一个问题也随之出现：**生成代码的速度，正在超过团队理解代码的速度。**

当一个改动能在几分钟内产生几百行代码时，人很容易把“它能运行”“测试通过”“AI 解释过了”当成“我们已经理解并接受了它”。代码一旦进入仓库，就属于未来要维护、排障和承担后果的人。

所以在 AI 时代，代码检视并没有过时。恰恰相反，它是团队保持项目控制权的关键机制。这也是 `explain-diff-for-human-review` 这个 Skill 出现的原因。

## 项目如何逐渐失控

代码风险不只有明显的 bug。项目失控往往以更安静的方式发生：新配置改变了默认行为，为复用增加的抽象模糊了原有边界，“临时”兼容逻辑成为永久负担，测试覆盖主流程却遗漏故障恢复，或者为了方便验证提交证书时把 CA 私钥也带进仓库。

这些改动可能有充分的局部理由，也未必会让测试失败。但如果团队没有看清系统影响，就会逐渐失去掌控：不知道某个行为为何存在，不敢删除多余逻辑，无法判断测试通过是否真的代表可以发布，事故发生后只能重新考古调用链。

AI 放大了这个问题，因为它极大降低了“再加一点代码”的成本。一句提示词就能生成新的分支、包装层、配置项和测试。代码变多不一定意味着能力同比增长，却一定意味着需要理解的表面积变大了。

真正需要守住的，不只是代码正确性，还有团队对代码的**可解释性、可追责性和可维护性**。

## 原始 diff 为什么不够

原始 diff 很精确，但它只告诉我们哪些行增加、删除或移动，通常不会主动回答：

- 这次修改要解决什么问题？
- 改动前后的系统模型有什么不同？
- 为什么选择这个职责边界？
- 用户和运维会观察到什么变化？
- 哪些配置仍然兼容？
- 测试证明了什么，又没有证明什么？
- 这里是缺陷、设计取舍，还是可接受的残余风险？

资深开发者可以沿着调用方、测试和历史提交还原答案。但当 diff 规模扩大，或者 reviewer 同时面对多个 AI 生成的改动时，这个过程会消耗大量注意力。

结果常常是：检视停留在命名、格式和局部实现上，真正的系统风险被跳过；或者 reviewer 因为理解成本太高，最终只确认“CI 是绿的”。问题不一定是 reviewer 不负责，而是输入材料不适合人建立完整心智模型。

## 我们不需要 AI 替人批准代码

让另一个 AI 自动检视，可以发现常见缺陷、危险 API 和缺失测试。但代码检视不只是找错，还包含项目所有者必须作出的判断：

- “任意一个集群可用就算 Ready”，是否符合可用性承诺？
- 为兼容旧配置保留一条路径，是否值得长期维护？
- 为本地验证提交固定证书，是否接受被误用的风险？
- 指标只有总量，是否足以支持生产排障？

这些问题没有脱离上下文的标准答案。AI 可以搜集证据、指出取舍、模拟失败场景，但最终决定必须由真正承担项目责任的人作出。

所以，`explain-diff-for-human-review` 的定位很克制：**它不替人检视，而是为人准备检视。**

## 这个 Skill 如何帮助人检视

Skill 可以接收单个提交、提交区间、分支比较、PR/MR、暂存区或工作区。它不会把 diff 逐行改写成自然语言，而是把改动整理成人可以作判断的材料。

### 还原系统模型

它会检查修改前的代码、调用方、测试、配置和文档，说明原有职责边界，再描述新的结构和数据流。

例如，一个代理组件（agent）从连接单个集群扩展为连接多个独立集群。真正需要确认的不是新增了几个字段，而是：一个地址代表实例还是集群入口？集群状态是否隔离？一个集群故障是否影响另一个？就绪状态（readiness）要求全部健康还是任意一个健康？

当这些关系被整理成修改前后模型时，reviewer 才能判断设计是否符合真实需求。

### 按行为组织改动

文件顺序通常不是最佳阅读顺序。一个功能可能同时修改配置、运行时编排、指标和测试。Skill 会按照“配置如何进入系统、状态如何维护、故障如何处理、指标如何反映、测试如何证明”的因果链组织内容，让人看到完整行为，而不是一组分散文件。

### 显式呈现影响与风险

报告会检查配置、协议、运行时、并发、安全、可观测性、部署、兼容性和测试，防止注意力只停留在局部实现。

它还会区分问题的性质：

- **已确认缺陷**：有明确失败路径，应在合入前修复；
- **设计决定**：实现可能正确，但需要项目负责人明确确认；
- **残余风险**：当前可以接受，但需要记录边界或后续处理；
- **提示**：帮助理解，不代表必须修改。

每项发现都包含证据、失败场景和处理建议。reviewer 不只看到“这里有风险”，还能知道风险如何发生，以及自己需要决定什么。

### 让验证证据可审计

Skill 严格区分本次实际执行的测试、diff 中存在但未执行的测试，以及建议补充的验证。它不会因为测试代码存在，就声称测试已经通过。

最终结果是一个自包含 HTML 文件，包含目录、架构图、影响矩阵、风险等级、可折叠证据和确认清单，可以直接打开、打印或留档。

选择 HTML 也是为了方便人检视。人毕竟是“视觉动物”：相比连续的文字和原始 diff，我们更容易通过清晰的层级、颜色、图表和空间关系理解复杂改动。HTML 可以把关键风险放到显眼位置，把细节折叠在需要时再展开，让 reviewer 更快建立整体认识，同时保留继续深挖证据的入口。

## 一个例子：能用，不代表适合合入

在一次真实检视中，最新提交只增加了本地验证需要的证书文件。它没有修改运行时代码，证书和私钥全部匹配，证书链也能通过验证。从“是否方便后续验证”看，它实现了目标。

但报告进一步发现：提交包含两把未加密 CA 私钥和四把端实体私钥，并且根证书是缺少标准 CA 扩展的 X.509 v1 证书。任何继续信任这些 CA 的环境，都可能被持有仓库内容的人伪造身份。

“是否方便验证”是事实；“是否接受公开测试 CA”“是否需要安全声明”“是否应修复证书生成流程”是项目决策。Skill 可以把事实和风险摆到同一张桌面上，但最终必须由人决定哪些代价可以接受。

## 人仍然负责项目走向哪里

这个 Skill 不能替代熟悉业务的人，也不能保证发现所有 bug。它无法知道团队尚未写下的承诺，更不能替负责人承担错误合入的后果。它所做的是组织证据、指出空白，让人更容易看见自己还不知道什么。

它不是裁判，而是一份更好的案卷。

AI 让人的角色从机械实现转向定义问题、选择边界、确认风险和维护系统一致性。只要代码仍然影响真实用户、真实数据和真实业务，我们就需要有人回答：

> 我们是否理解这次改变？是否接受它的行为和代价？如果半年后出现问题，是否知道从哪里开始处理？

AI 可以帮助我们写得更快。检视让我们仍然知道，项目正在走向哪里。
